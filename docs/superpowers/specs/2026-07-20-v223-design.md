# V223 — Prix progressif + parcours « payer d'abord »

> Date : 20 juillet 2026
> Statut : spec validée, implémentation non commencée
> Règle absolue : **tout est additif**. Aucun code existant supprimé ou modifié dans son comportement.

---

## 1. Contexte et objectif

Aligner le parcours client d'afroboost sur celui de spordateur.com :

- le client voit une activité avec **3 paliers de prix** (Early Bird / Standard / Last Minute) ;
- il paie **d'abord** sur Stripe, sans remplir de formulaire ;
- il reçoit son code `AFR-XXXXXX` par email ;
- il choisit ses dates **après**, dans `/espace/AFR-XXXXXX`, avec ses crédits.

Le parcours actuel (formulaire Nom / Email / WhatsApp / dates → PAYER ET RÉSERVER) reste
**strictement inchangé** pour toutes les offres sans prix progressif.

---

## 2. Découvertes d'exploration ayant modifié le périmètre

Le brief initial décrivait 8 chantiers. L'exploration du code a montré que la moitié
existait déjà ou visait la mauvaise collection. Périmètre final : **4 changements**.

### 2.1 `Course` n'est pas l'objet commercial

`api/server.py:336-349` — `Course` est un **créneau hebdomadaire récurrent** :

```python
class Course(BaseModel):
    weekday: int      # 0-6
    time: str         # "19:00"
    locationName: str
    # ... aucun prix, aucune date
```

Un cours n'a **ni prix ni date unique**. Le prix progressif a besoin d'une date de
référence fixe pour calculer « 7 jours avant ». Sur un créneau récurrent, le prix
repartirait à zéro chaque semaine (cher le dimanche, bradé le lundi) — comportement
incompréhensible pour l'acheteur et source de demandes de remboursement.

`Offer` (`api/server.py:363-401`) porte en revanche `price`, `linked_course_ids`, et
surtout `countdown_enabled` / `countdown_date` / `countdown_time` (v132) — le compte à
rebours déjà en production.

**Décision : le prix progressif vit sur `Offer`, la date de référence est
`countdown_date`.** `Course` n'est pas touché.

Bénéfice : le compte à rebours affiché et le palier tarifaire actif dérivent de la
**même** date, donc restent cohérents sans synchronisation supplémentaire.

### 2.2 La création de pack existe déjà (chantiers 2C et 2E abandonnés)

`api/server.py:3624-3849`, branche `else` du webhook `checkout.session.completed`, fait
déjà **l'intégralité** de ce que demandaient les chantiers 2C et 2E :

| Étape demandée | Déjà implémenté |
|---|---|
| Générer `AFR-XXXXXX` | `server.py:3660` |
| Insérer dans `discount_codes` (`type: "100%"`, `maxUses`) | `server.py:3663` |
| Insérer dans `subscriptions` | `server.py:3707` |
| Email client + QR code | `server.py:3710-3820` |
| Email coach « ⭐ Nouvelle souscription » | `server.py:3825-3848` |
| Upsert contact CRM | `server.py:3823` |

Réimplémenter ces chantiers produirait **deux codes AFR et deux emails par achat**.
C'est le principal risque de régression du brief initial.

### 2.3 Les trois webhooks Stripe ne sont pas en conflit

Hypothèse initiale de shadowing **invalidée**. Trois chemins distincts :

| Handler | Chemin réel | Rôle |
|---|---|---|
| `server.py:3464` | `/api/webhook/stripe` | **principal** — souscriptions |
| `stripe_routes.py:363` | `/api/stripe/webhook` | crédits coach uniquement |
| `checkout_routes.py:445` | `/api/checkout/webhook/stripe` | vitrine uniquement |

Seul le premier est modifié par V223.

### 2.4 Bug live : les crédits sont déduits d'une regex sur le nom du produit

`server.py:3649-3659` :

```python
x_match = _re_webhook.search(r'x\s*(\d+)', product_name, ...)
# repli : "10" in product_name / "5" in product_name
```

« Pack 5 séances » fonctionne par chance. **« Cours du 10 mai » accorde 10 crédits.**
Le champ `pack_sessions` corrige ce bug à la racine.

### 2.5 La collection `contacts` n'existe pas

Le CRM lit `chat_participants` (`server.py:3823`, `6413`, `7573`). Écrire dans une
nouvelle collection `contacts` rendrait les données marketing **invisibles au dashboard**.

**Décision : l'onboarding écrit dans `chat_participants`.**

### 2.6 Pas de react-router, et `PaymentSuccessPage` a été désactivée volontairement

Aucun `<Route>` dans `App.js` — le routage est du pattern-matching manuel
(`App.js:2981`). Le retour de paiement passe par query params (`App.js:3172-3210`).

`PaymentSuccessPage.jsx` existe mais est neutralisée en v158.7 :

```js
{false && showPaymentSuccessPage && lastReservation && (   // App.js:4812
```

**Décision : réutiliser le mécanisme query-param existant ; réactiver
`PaymentSuccessPage` uniquement derrière le flag `progressive_pricing`.** La décision
v158.7 reste donc valable pour 100 % des offres actuelles.

## 4. Backend

### 4.1 `compute_active_price(offer) -> dict`

Fonction **pure** (aucun accès DB), donc testable unitairement.

Date de référence : `countdown_date` + `countdown_time`.

Retour : `{"price": float, "tier": str, "original_price": float, "currency": "CHF"}`

Échelle de dégradation :

| Condition | `tier` | `price` |
|---|---|---|
| `progressive_pricing` faux ou absent | `regular` | `offer.price` |
| Progressif mais `countdown_date` absent | `regular` | `offer.price` |
| > `early_bird_days_before` jours avant | `early_bird` | `price_early_bird` ou `offer.price` |
| > `standard_hours_before` heures avant | `standard` | `price_standard` ou `offer.price` |
| Sinon (y compris date dépassée) | `last_minute` | `price_last_minute` ou `offer.price` |

Chaque palier retombe sur `offer.price` si son prix n'est pas défini : une configuration
partielle ne peut jamais produire un prix nul ou vide.

**Sans `countdown_date`, on retombe sur le prix normal** — jamais sur Early Bird. Afficher
le palier le moins cher par défaut ferait vendre au tarif le plus bas toute offre mal
configurée. Le dashboard affichera l'indication : « Activez le compte à rebours pour que
les paliers s'appliquent. »

### 4.2 Lectures

- `GET /api/offers` : les réponses **gagnent** `active_price` et `active_tier`.
  Aucun champ retiré ni renommé. Permet à la page d'accueil d'afficher les paliers sans
  requête supplémentaire par carte.
- `GET /api/offers/{offer_id}/active-price` : nouvel endpoint pour la page activité.

### 4.3 Checkout (`server.py:3330`)

Ajout de `pack_sessions` et `tier` dans les metadata Stripe. Les metadata existantes
sont conservées telles quelles.

### 4.4 Webhook (`server.py:3634-3659`) — deux modifications chirurgicales

**a) Email client — correctif bloquant**

Ligne actuelle :

```python
customer_email = session.get("customer_email") or metadata.get("customer_email", "")
```

Sans formulaire préalable, c'est Stripe qui collecte l'email — et Stripe le place dans
`session.customer_details.email`, **pas** dans `session.customer_email`. Les deux termes
du `or` seraient donc vides : souscription créée avec un email vide, aucun email envoyé,
client sans code et sans accès à `/espace/` après avoir payé.

```python
# V223: l'email saisi sur Stripe doit primer, sinon l'acheteur ne reçoit rien
customer_email = (session.get("customer_details") or {}).get("email") \
    or session.get("customer_email") or metadata.get("customer_email", "")
```

Modification en **élargissement de repli uniquement** : l'ordre des deux termes existants
est préservé, donc le flux formulaire actuel est inchangé.

**b) Nombre de crédits**

```python
# V223: faire confiance au champ ; regex conservée pour les offres legacy
sessions_count = int(metadata["pack_sessions"]) if metadata.get("pack_sessions") \
    else <regex existante inchangée>
```

### 4.5 `PUT /api/subscriptions/{code}/profile`

Corps : `{name, whatsapp}`. Effets : mise à jour du document `subscriptions`, puis upsert
dans `chat_participants` avec `source: "espace_onboarding"`.

## 5. Frontend

### 5.1 `OffersManager` — section prix progressif

Après le champ prix existant : toggle + 3 champs numériques + lien « Réinitialiser aux
valeurs suggérées » (Early Bird = prix de base, Standard = ×1.33, Last Minute = ×2).
Champ `pack_sessions` séparé.

Style : fond noir, bordures `rgba(217,28,210,0.2)`, accents `#D91CD2`, texte blanc,
toggle rose/violet actif.

### 5.2 `App.js` — page activité

Si `progressive_pricing` : bannière, compte à rebours existant, bouton rose
« Réserver — X CHF → » avec le palier actif, puis les 3 cartes de paliers (bordure rose
sur le palier actif) et les détails DATE / DURÉE / LIEU / PLACES.

Le bouton redirige **directement vers Stripe Checkout**, sans formulaire.

Sinon : le formulaire actuel s'affiche, à l'identique.

### 5.3 `OfferCardSlider`

Si `progressive_pricing` : prix du palier actif + badge (`🎯 Early Bird` vert,
`Standard` jaune, `🔥 Last Minute` rouge), bouton « Réserver — X CHF ».

### 5.4 `SubscriberOnboarding.js` (nouveau fichier)

`SubscriberSpace.js` — fichier de 57 Ko utilisé par tous les abonnés payants — ne reçoit
qu'un **import et une condition**. Toute la logique du formulaire vit dans le nouveau
composant, isolé et remplaçable.

Déclenchement : `name` ou `whatsapp` absent du document `subscriptions`. Affichage **une
seule fois**.

**Lien « Plus tard » obligatoire.** Sans échappatoire, tout abonné existant dont la fiche
n'a pas ces champs — ils n'ont jamais été obligatoires, donc il y en aura — se
retrouverait définitivement enfermé hors de crédits déjà payés.

### 5.5 `PaymentSuccessPage.jsx`

Réactivée **uniquement** si l'offre achetée a `progressive_pricing`. Affiche le code et
un bouton « Réserver mes dates → » vers `/espace/{code}`. Le `false &&` de la v158.7 reste
en place pour tout le reste.

Restent intacts : codes `AFR-` existants, liens de conversation, chat, paiements
carte / TWINT / PayPal, formulaire actuel, les 53 routes JWT, le Service Worker.

## 8. Hors périmètre — à traiter séparément

**Le webhook principal ne vérifie pas la signature Stripe** (`server.py:3469`) :

```python
event = stripe.Event.construct_from(stripe.util.json.loads(body), stripe.api_key)
```

`construct_from` ne valide rien. Quiconque connaît l'URL peut POSTer un faux
`checkout.session.completed` et se générer un pack `AFR-` valide, avec le nombre de
crédits de son choix, sans payer.

Correctif : `stripe.Webhook.construct_event(body, sig_header, STRIPE_WEBHOOK_SECRET)`.

Volontairement **exclu de V223** : mêler un correctif de sécurité à une livraison
fonctionnelle rend les deux plus difficiles à relire et à annuler. À livrer en commit
dédié.
